# N+1 발생 상황

## 문제 발생 시나리오
주문 목록을 페이징 조회할 때 N+1문제가 발생하였습니다.
```java
Page<Order> orderPage = orderRepository.findAll(pageable);
```

### 원인 분석
Order 엔티티에 orderLines 컬렉션이 @ElementCollection과 FetchType.EAGER로 설정되어있습니다.
```java
@ElementCollection(fetch = FetchType.EAGER)
@CollectionTable(name = "tb_order_line", joinColumns = @JoinColumn(name = "order_id", comment = "주문ID"))
@OrderColumn(name = "line_seq", columnDefinition = "INT COMMENT '주문 항목 순서'")
private List<OrderLine> orderLines;
```
이 설정으로 인해 다음과 같은 쿼리 실행 패턴이 발생합니다.

### 실행되는 쿼리
1. 1번의 Order 조회 쿼리(페이징 적용)
``` sql
    select
        o1_0.id,
        o1_0.order_no,
        o1_0.order_state,
        o1_0.receiver_name,
        o1_0.receiver_phone,
        o1_0.zip_code,
        o1_0.address1,
        o1_0.address2 
    from
        tb_order o1_0 
    order by
        o1_0.id desc 
    limit
        ?, ?
```

2. N번의 OrderLine 조회 쿼리(조회된 주문 개수만큼)
```sql
select
        ol1_0.order_id,
        ol1_0.line_seq,
        ol1_0.order_amount,
        ol1_0.order_currency,
        ol1_0.order_price,
        ol1_0.product_id,
        ol1_0.quantity 
    from
        tb_order_line ol1_0 
    where
        ol1_0.order_id=?
```
총 쿼리 실행 횟수: 1+N

### 문제점
- 조회된 주문(Order)이 10개라면 총 11번의 쿼리 실행
- 데이터베이스 조회 횟수가 증가하여 성능 저하 발생
- @ElementCollection은 별도 엔티티가 아닌 값 타입 컬렉션이므로 Fetch Join을 사용할 수 없어 문제가 더 심각함.
 
### 해결 방안
1. LAZY로딩으로 변경하고, Batch Fetch Size 설정 ❌ 근본적인 해결방법이 아님.
2. @EntityGraph 사용 ❌ 페이징+컬렉션 fetch join 시 in-memory pagination 발생.
3. ElementCollection -> OneToMany로 변경 ❌ OrderLine은 밸류 타입임. 
4. DTO 프로젝션 활용 ⭕
5. @ElementCollection의 FetchType을 LAZY로 변경하여, 컬렉션을 필요할 때만 조회하도록 함. 
   리스트 조회용 DTO(OrderListResponse)에는 OrderLine을 포함하지 않고, 필요한 Order 정보만 매핑하여 반환함.
   ❌ EAGER → LAZY로 바꾸면 조회 최적화를 위해 도메인 모델의 설정을 변경하는 것이라, Command 쪽에서 OrderLine이 필요한 경우 매번 추가 조회가 발생하는 부작용이 생김.
       조회 관심사가 도메인 모델을 오염시킴.

### 최종 해결책
CQRS 패턴 도입하여 Command와 Query 책임을 분리한다.
Command에서는 도메인 모델을 중심으로 상태 변경과 비즈니스 규칙을 처리하고,
Query에서는 조회 성능과 데이터 전달을 최적화하기 위해 DTO에 쿼리 결과를 바로 담아 반환한다.
조회 시 불필요한 엔티티 로딩을 방지하고 N+1 문제를 피할 수 있다.
✅ 주문목록 조회 시 응답을 위한 OrderListDto를 생성하고 쿼리 결과를 DTO에 바로 담아서 반환하여 N+1문제 해결하였다.
